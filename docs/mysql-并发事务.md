# mysql并发事务

我们用一张 user 表举例，初始数据：id=1, money=100
![](img\mysql-并发事务.png)

```text
为什么不会报错 / 阻塞？


在 RR 隔离级别下,
普通 UPDATE 语句 ：
    普通查询（快照读）
    事务 A/B 一开始的查询，读的是事务启动时的快照
    彼此看不见对方的修改
更新操作（当前读）
    执行 UPDATE 时，不会用快照里的旧值 (更新/删除/SELECT FOR UPDATE：读当前最新真实数据, 即 强制读MVCC的最新版本+加锁💥💥💥)
    会立刻读取数据库当前最新值，并加锁更新
    所以事务 B 更新时，能看到事务 A 已经提交的 200，并直接覆盖成 300
```
```text
什么情况下事务 B 会失败 / 阻塞？


只有两种场景会出问题：
使用 SELECT ... FOR UPDATE（加锁查询）
    事务 B 先查询并加锁 → 事务 A 更新会阻塞等待
使用 UPDATE ... WHERE 旧值 这种乐观锁写法
    例如：UPDATE user SET money=300 WHERE id=1 AND money=100;
    事务 B 更新时，真实值已经是 200，条件不匹配，更新 0 行
```
