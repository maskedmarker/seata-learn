# AT全局事务(Auto Transaction)


## 全局事务完整提交流程

```text
从全局事务入口开始,讲一下整个事务的提交过程
我用最清晰、最通俗、一步不落的方式，给你讲一遍 Seata AT 模式从全局事务开启 → 执行 → 提交 / 回滚 的完整流程。只讲流程原理，不讲废话，你一看就能懂。

全局事务完整提交流程（从入口到最终提交）
环境前提：
TM：订单服务（加了 @GlobalTransactional 的方法）
RM1：库存服务
RM2：账户服务
TC：Seata Server


0. 准备工作（启动时就做好）
三个微服务启动，自动向 TC 注册自己为 RM
数据源被 Seata 自动代理：DataSourceProxy
每个库都有 undo_log 表，用于记录前镜像 + 后镜像



1. 全局事务开启（TM 发起）
入口方法执行：
@GlobalTransactional
public void createOrder() {//...}

TM 做的事：
1. TM 向 TC 发送请求：开启一个全局事务
2. TC 生成一个全局唯一 ID：XID
3. TC 返回 XID 给 TM
4. TM 把 XID 放入当前线程上下文（RootContext）
此时：全局事务已创建，状态为 Begin


2. 执行业务逻辑（本地事务 + 远程调用）
TM 开始执行：
1. 创建订单（本地操作）
2. 远程调用库存服务 deduct
3. 远程调用账户服务 deduct
关键点：Feign/RestTemplate 会自动把 XID 放到请求头传到下游服务，下游 RM 拿到 XID，就知道自己属于哪个全局事务。


3. 每个 RM 执行分支事务（核心：AT 两阶段）
每个 RM（库存、账户、订单）执行 SQL 时，流程完全一样：
阶段一：执行 SQL + 生成 undo_log
1. RM 开启本地数据库事务
2. 执行业务 SQL（update stock /update account）
3. Seata 拦截 SQL，生成：
    before image（前镜像）：修改前的数据
    after image（后镜像）：修改后的数据
4. 把镜像插入 undo_log
5. RM 本地事务提交（注意：这里本地已经提交了！）
6. RM 向 TC 注册分支事务：上报分支 ID、XID、状态
重点：AT 模式一阶段本地就提交了，不是等全局提交才提交。靠 undo_log 保证可回滚。



4. 所有分支执行完成，TM 发起全局提交
当业务代码执行完，没有抛异常，TM 开始做全局提交。
TM 向 TC 发送：GlobalCommitRequest(XID)


5. TC 收到全局提交，开始协调所有 RM
TC 遍历该 XID 下所有注册的分支事务：
●库存分支
●账户分支
●订单分支
TC 给每个 RM 发送：BranchCommitRequest(XID, branchId)



6. RM 收到全局提交请求，做清理工作
RM 收到 TC 的提交指令：
1. 根据 XID + branchId 找到对应的 undo_log
2. 删除这条 undo_log
3. 向 TC 返回：分支提交成功
提交阶段 RM 什么都不用回滚，也不用改数据，只需要清理日志。



7. 所有分支提交完成 → TC 结束全局事务
1. TC 收集所有 RM 的 BranchCommitSuccess
2. TC 将全局事务状态改为 Committed
3. TC 清理全局事务记录
4. TM 方法正常结束
最终结果
订单插入成功
库存扣减成功
余额扣减成功
undo_log 被清空
全局事务完成



一张图总结提交流程（极简版）
TM(@GlobalTransactional)
    ↓ 申请XID
TC(Server)
    ↓ 返回XID
TM 执行业务 + 传递XID到下游
    ↓
各RM执行SQL
    ↓
RM本地提交 + 写undo_log
    ↓
RM注册分支到TC
    ↓
所有分支执行完毕
    ↓
TM请求TC全局提交
    ↓
TC通知所有RM提交
    ↓
RM删除undo_log
    ↓
TC标记全局事务提交完成



反过来：如果抛异常，回滚流程是什么？
为了让你彻底理解，我顺便把回滚流程也极简讲完：
1. TM 抛异常
2. TM 向 TC 发送 GlobalRollback
3. TC 向所有 RM 发送 BranchRollback
4. RM 根据 undo_log 前镜像还原数据
5. RM 删除 undo_log
6. TC 标记全局事务已回滚
```

## undo_log 到底怎么实现回滚的

前镜像 = 回滚的目标状态
后镜像 = 用来校验有没有被别人动过
回滚 = 根据前镜像生成反向 SQL，把数据改回去

为什么必须主键？
因为 Seata 回滚时要精准定位那一行，必须用主键定位，否则无法安全回滚。

```text
1. 先看 undo_log 里存了什么
Seata 在执行你的业务 SQL（update/insert/delete）时，会自动查一遍数据，生成两个东西：
beforeImage（前镜像）执行 SQL之前的数据长啥样
afterImage（后镜像）执行 SQL之后的数据长啥样
这两个镜像，会被序列化后存到 undo_log.rollback_info 里。同时还存：
xid
branchId
表名、主键值


2. 一阶段执行时（正常执行）
以扣库存为例：update stock set stock = stock - 1 where product_id = 1;

Seata 做了 4 步：
1. 查询前镜像执行前先查一遍：select * from stock where product_id=1→ 存为 beforeImage
2. 执行你的 update库存从 10 → 9
3. 查询后镜像再查一遍→ 存为 afterImage
4. 插入 undo_log把 before + after 存进去然后本地事务提交！
重点：业务数据已经提交了，不是锁住的。靠 undo_log 保证可回滚。



3. 回滚时到底干了啥？（核心）
当全局事务要回滚，TC 通知 RM：branchRollback(xid, branchId)
RM 拿到后，做这几件事：
步骤 1：读取 undo_log
根据 xid + branchId 找到那条日志，反序列化得到：
    beforeImage
    afterImage
    表名
    主键

步骤 2：校验当前数据是否被别人篡改（重点机制）💥💥💥
Seata 不会直接无脑回滚！它会现在再查一遍当前数据，和 afterImage 对比：
    如果一样 → 说明没人改过，可以安全回滚
    如果不一样 → 说明数据被其他事务修改了，出现脏回滚，直接抛异常，需要人工处理
这就是 Seata 的 幂等 + 防悬挂 + 防脏回滚 机制。

步骤 3：用 beforeImage 生成回滚 SQL
Seata 自动拼 SQL：
① 如果原操作是 UPDATE
回滚 SQL 就是：update stock set stock = #{before.stock}, ...其他字段 where id = #{before.id}
直接把数据还原成执行前的样子。
② 如果原操作是 INSERT
回滚就是：delete from stock where id = #{before.id}
③ 如果原操作是 DELETE
回滚就是：insert into stock (...) values (...)

步骤 4：执行回滚 SQL
执行完，数据恢复如初。

步骤 5：删除 undo_log
回滚完成，这条日志没用了，删掉。
```

## 全局锁

全局锁要等到全局事务二阶段提交或回滚完成才释放，因此同一行数据的多个全局事务会串行执行。

```text
核心结论
1. 多个全局事务之间，本身是并行的，不会互相等待
2. 但如果它们同时修改同一行数据（比如同一件商品库存）
    → Seata 全局锁 + 数据库行锁 会强制它们排队执行
3. 最终效果：
    同一行数据，同一时间，只有一个全局事务能修改
    其他全局事务必须等待锁释放，才能继续
下面我一步步拆开讲，你看完就完全通透。


0. 先明确两个锁的作用
数据库行锁（innodb lock）
    保护本地事务不脏写，是数据库层面的锁。
Seata 全局锁（Global Lock）
    保护全局事务回滚时不被污染，是 Seata TC 层面的锁。
AT 模式要安全，两个锁必须一起用。


1. 场景：两个并发请求，同时扣同一个库存
请求 1：用户 A 买商品 1
请求 2：用户 B 买商品 1
库存表：product_id=1，stock=10
两个请求同时进入：createOrder()
它们是两个不同的全局事务：
XID-1
XID-2


2. 关键：扣库存时，Seata 做了什么？
Seata AT 会做 3 件非常关键的事：
① 先获取 Seata 全局锁
RM 向 TC（Seata Server）发起：
申请全局锁：resource=stock, pk=1, xid=?
结果：
XID-1 先抢到锁 → 成功
XID-2 抢不到 → 阻塞等待
② 获取成功，才执行数据库 update
此时才真正执行 SQL：update stock ... where product_id=1
③ 本地事务提交，释放数据库行锁
生成 before/after 镜像
插入 undo_log
本地事务提交
→ 释放 MySQL 行锁
但注意：Seata 全局锁此时 不释放！



3. 重点：全局锁什么时候释放？
Seata 全局锁 要等到全局事务 二阶段完成 才释放：
两种情况：
情况 A：全局事务提交
    TM 通知 TC 提交
    TC 异步通知所有 RM 删除 undo_log
    TC 释放全局锁
    此时，另一个事务 XID-2 才能抢到锁
情况 B：全局事务回滚
    TM 通知 TC 回滚
    RM 根据 undo_log 恢复数据
    TC 释放全局锁
    下一个事务才能继续
    
    
4. 最终表现：同一行数据，全局事务串行执行
你看到的最终效果是：
plaintext
请求1 开始执行
请求1 扣库存（拿全局锁）
请求1 本地提交完成
请求2 一直阻塞在“扣库存”这一步
请求1 全局提交/回滚 → 释放全局锁
请求2 才开始扣库存    



5. 为什么要这么设计？（防脏回滚）
如果不锁，会发生灾难：
事务 1 扣库存：10 → 9
本地提交，行锁释放
事务 2 立即扣库存：9 → 8
事务 1 报错，要回滚，把库存恢复成 10
→ 超卖 + 数据错乱
Seata 全局锁就是为了禁止这种情况：只要全局事务没结束，任何人不能改这行数据。
```

```text
全局锁保存在哪里?

一、核心结论
Seata AT 全局锁，只保存在 TC（Seata Server）端，不在 RM（业务服务）、也不在业务数据库里；内存为主、持久化为辅。

二、全局锁的核心存储位置（分两层）
1. 主存储：TC 内存（HashMap，读写最快）
    所有全局锁优先存在 TC 进程内存，以 rowKey -> XID 的哈希表（ConcurrentHashMap）维护。
    rowKey 格式：资源ID + 表名 + 主键值（比如 jdbc:mysql://xxx:3306/seata:stock:1），精准锁定单行数据。
    作用：锁申请 / 释放 / 冲突判断 全在内存完成，毫秒级响应，保证高并发性能。
    
2. 持久化存储（防止 TC 宕机丢锁）
根据 Seata 配置 store.lock.mode（1.5.1+ 支持锁与会话分离），有 4 种持久化方案：    
    file（默认，单机）：本地文件 root.data，不支持集群高可用。
    db（生产集群常用）：MySQL 表 lock_table（必须手动建表），字段：row_key（主键）、xid、branch_id、table_name、pk 等。
    redis（高性能集群）：Redis 哈希 / 字符串存储，支持高可用、性能优于 DB。
    raft（Seata 2.x+）：基于 Raft 日志同步到所有 Follower 节点内存，无独立持久化文件 / 表，强一致、高可用。
    
    
三、关键细节
1. 锁归属：只在 TC，不在 RM
    RM（订单 / 库存 / 账户服务）不存全局锁，只负责向 TC 申请 / 释放锁、执行本地事务、写 undo_log。
    业务数据库只有 undo_log、业务表，没有全局锁数据。    
2. 锁生命周期：内存持有，持久化备份
    申请锁：RM 注册分支事务 → TC 内存加锁 → 同步写入持久化（file/db/redis）。
    持有锁：全局事务一阶段（本地提交）完成后，内存锁一直持有，直到二阶段提交 / 回滚（这就是为什么要等全局事务结束才释放）。
    释放锁：TC 完成二阶段 → 内存删除锁 → 持久化删除对应 row_key 记录。    
3. 集群一致性
    单机 TC：内存 + 本地 file/db/redis。
    集群 TC（db/redis/raft）：所有节点共享同一份持久化锁数据，保证全局唯一锁判断。    
    
一句话再总结
全局锁 = TC 内存哈希表（主） + TC 持久化（file/db/redis/raft，备），只在 Seata Server，不在业务服务 / 业务库。    
```


## 在创建订单时, 扣库存发生了异常, seata会发生什么?

```text
在创建订单时, 扣库存发生了异常, seata会发生什么?


场景固定
订单服务（TM）：
@GlobalTransactional
public void createOrder() {
    // 1. 本地创建订单（已执行、已插入库）
    orderMapper.insert(order);

    // 2. 远程调用库存服务扣减 → **这里抛异常了！**
    stockClient.deduct(); // 抛出 Exception
}
现在：扣库存这一步报错了


下面是 Seata 完整连锁反应。
一、第一步：异常抛到 TM（订单服务）
1. 库存服务（RM）抛异常
2. Feign 把异常抛回订单服务
3. 异常穿过 @GlobalTransactional 切面
TM 立即做两件事：
    标记当前全局事务需要回滚
    立刻向 TC（Seata Server）发起全局回滚请求：GlobalRollbackRequest(XID)
    
二、第二步：TC 收到回滚指令，开始协调所有 RM
TC 内部记录了这个 XID 下所有分支事务：
分支 1：订单服务（order）
分支 2：库存服务（stock）
TC 会遍历所有分支，依次发送：BranchRollbackCommand(xid, branchId)   

三、第三步：各个 RM 执行回滚（核心） 

四、第四步：所有分支回滚完成，TC 结束全局事务

五、最终结果（你查库看到的）
    订单表：没有新增订单
    库存表：数量没变回去了
    undo_log：全部被清空
    全局锁：全部释放
数据完全一致，像什么都没发生过。


最关键的一张流程图
扣库存抛异常
    ↓
异常传到 TM（订单服务）
    ↓
TM 请求 TC 全局回滚
    ↓
TC 通知所有分支（订单、库存）回滚
    ↓
每个RM根据undo_log前镜像恢复数据
    ↓
RM删除undo_log
    ↓
TC释放全局锁
    ↓
全局事务回滚完成


Q5：如果扣库存异常，账户还没调用，会怎样？
账户分支根本没注册，TC 不会管它，只回滚已经执行的分支。
```

## 连接超时

### TC <-> RM 断网
```text
一句话结论:
TC 跟 RM 网络断了/超时 → TC 不会无限等，会走：超时检查 + 重试 + 最终事务状态裁决。



场景
全局事务已经一阶段完成：
    订单已插入（已提交）
    库存已扣减（已提交）
    账户已扣减（已提交）
现在 TM 让 TC 提交 / 回滚，但 TC 通知 RM 时，网络断了 / 连不上 RM。


一、先明确：TC 怎么跟 RM 通信？
TC 对 RM 下发指令有两种：
    Branch Commit（提交：删 undo_log）
    Branch Rollback（回滚：恢复数据）
通信方式：Netty 长连接 + 重试机制



二、网络不通 / 超时，TC 会做什么？
1）立即重试（默认 3 次）
发一次不通/等一会儿再发/再不通再发
目的：大概率是临时抖动。
2）重试依然失败 → TC 标记该分支为 超时 / 失败
TC 不会因为一个 RM 连不上，就卡死整个全局事务。
3）TC 启动 定时任务（TransactionTimeoutChecker）
后台线程不断轮询：
    全局事务是否超时
    分支事务是否长期未响应
4）达到阈值后，TC 做最终裁决
裁决规则非常关键，分两种情况：

三、情况 1：全局事务是 回滚（Rollback）
例如扣库存异常，需要回滚。
TC 连不上 RM 怎么办？
    TC 会不断重试回滚指令
    重试失败 → 把该分支标记为 RollbackRetrying
    继续后台定时重试（频率越来越低，但一直试）
    直到 RM 重新连上，TC 再次下发回滚
最终保证:必须回滚成功，数据必须恢复。只要 RM 重新连上，就一定能回滚。



四、情况 2：全局事务是 提交（Commit）
所有服务都正常，TC 要下发 BranchCommit。
如果 TC 连不上 RM：
    TC 同样重试
    失败 → 标记为 CommitRetrying
    后台继续定时重试
    RM 重连后，TC 再次下发提交指令
RM 收到 Commit 做什么？
只干一件事：删除对应的 undo_log不影响业务数据，因为业务数据一阶段已经提交了。
极端情况：一直连不上，会怎样？
业务数据不受影响，已经正常提交。只会残留一条 undo_log，不会脏数据、不会超卖。等 RM 恢复后，TC 补发指令，RM 删掉日志即可。



五、最关键机制：TC 不会阻塞，也不会 “不确定”
Seata 设计原则：
    一阶段成功 = 业务数据已落地
    二阶段（提交 / 回滚）= 日志操作，可重试、可异步、可最终一致
所以：
    网络问题不会导致业务数据错乱
    只会导致 undo_log 暂时没删，或回滚暂时没执行
    不会出现 “一半提交一半回滚”
    
    
六、官方真实处理流程
TC 下发 BranchCommit / BranchRollback
网络超时 → 重试 3 次
仍失败 → 事务状态标记为 重试中
后台 定时任务 持续重试
RM 重连后自动恢复执行
最终一定能到达：
    提交：删除 undo_log
    回滚：恢复数据 + 删除 undo_log    
    
    
七、灵魂问题
Q：如果 RM 永远连不上了（机器炸了），怎么办？
Seata 不会死等。
    业务数据已经在一阶段提交
    若为提交场景：数据正确，仅残留 undo_log
    若为回滚场景：等 RM 重启后，TC 会继续推送回滚指令
    人工也可以通过 undo_log 手动回滚
Q：会不会出现 “数据悬着” 的情况？
不会。一阶段已经确定数据状态，二阶段只是日志清理或补偿。    
```

### TM <-> TC 断网

```text
这个问题比 TC↔RM 断网 更关键，也更容易出现事务悬挂、数据不一致，我给你讲最标准、最真实、生产级的处理逻辑，一步到位讲透。


先说结论
TM ↔ TC 网络断了 → 会出现两种典型场景：
    TM 开启全局事务时断网 → 根本开不起来，直接失败
    TM 一阶段执行完，通知 TC 提交 / 回滚时断网 → 最危险，会产生 “事务悬挂”, TC 有超时裁决机制，超时后自动回滚，避免锁一直占用
    

场景 1：TM 刚要开启全局事务，连不上 TC
此时断网 / 超时：
    TM 获取 XID 失败
    直接抛异常
    整个全局事务不启动
    一行业务 SQL 都不会执行
结果：完全安全，无任何数据问题。    


场景 2：一阶段已执行完，TM 要通知 TC 提交 / 回滚时断网
这是最核心、最危险的场景！

执行链路
    TM 已拿到 XID，全局事务已开启
    订单、库存、账户都执行完成
        → 本地全部提交，undo_log 已写，全局锁已持有
    业务代码正常结束 / 抛出异常
    TM 准备通知 TC：提交 or 回滚
就在这时：TM ↔ TC 网络断开！
此时状态极其关键：
    RM 们已经一阶段完成
    业务数据已经提交
    全局锁还在 TC 里被持有
    TC 还在等 TM 的最终指令    
    
TM 这边会做什么？
1. TM 尝试发送 GlobalCommit/Rollback
2. 网络不通 → 抛异常
3. TM 本地线程结束
4. 但 TM 没有任何机制 “记住” 这个事务
5. TM 不持久化事务日志，断了就忘了
这就会出现一个经典问题：TM 忘了，但 TC 还在等，全局锁还在占着！   这就叫 事务悬挂（Hanging Transaction）    

TC 怎么处理？（核心：超时裁决）
TC 不知道 TM 是挂了还是网络断了，所以 TC 只遵守一个规则：全局事务超时 → 默认强制回滚！💥💥💥
流程：
1. TC 有一个后台定时任务：GlobalStatusCheckTask
2. 每隔一段时间扫描所有进行中的全局事务
3. 如果某个 XID 超过超时时间（默认 60s）还没收到 TM 的 commit / rollback
TC 直接裁决：强制回滚该全局事务！

TC 强制回滚后做什么？
1. TC 向所有已注册的 RM 发送 BranchRollback
2. RM 根据 undo_log 恢复数据
3. 删除 undo_log
4. 释放全局锁
5. 全局事务状态改为 Rollbacked
最终结果：
    订单被删
    库存恢复
    余额恢复
    全局锁释放
    数据一致
    
为什么必须 “超时默认回滚”？
为了防止死锁、防止全局锁长期占用。
如果超时不回滚：
    热点库存行全局锁一直被占
    其他所有下单请求全部阻塞
    系统直接卡死
所以 Seata 的设计哲学非常明确：在不确定时，选择安全：回滚。    💥💥💥


极端情况：TC 回滚时，TM 网络又恢复了
会出现：
    TM 后来又发了个 commit
    TC 已经把事务回滚了
TC 怎么处理？
直接拒绝：TransactionAlreadyRolledBack
保证不会重复提交、不会数据错乱。



面试标准答案
问：TM 与 TC 之间断网，Seata 如何处理？
答：
    TM 开启事务时断网 → 直接失败，不执行业务。
    一阶段完成后断网 → TM 无法通知 TC 提交 / 回滚。
    TC 有全局事务超时检查机制，默认 60s 超时。
    超时未决的事务，TC 强制自动回滚。
    回滚所有 RM 分支、释放全局锁，保证数据一致。
    目的是避免事务悬挂、全局锁长期占用导致系统阻塞。
    
    
如果 TC 也挂了怎么办？
TC 重启后，会从存储（db/file/redis）加载所有未完成事务，继续超时检查 → 依然回滚。    
```
